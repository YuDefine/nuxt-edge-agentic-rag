## Requirements

### Requirement: Better-auth Tables Storage Type Matches Drizzle Declaration

The D1 column type for every better-auth-owned table (`user`, `account`, and any other tables generated by `.nuxt/better-auth/schema.sqlite.ts`) SHALL match the drizzle column declaration at runtime. Specifically, columns declared as `integer({ mode: 'timestamp_ms' })` — including `user.createdAt`, `user.updatedAt`, `user.banExpires`, `account.createdAt`, and `account.updatedAt` — SHALL be stored with SQLite `INTEGER` affinity in the live D1 database, so that drizzle's `timestamp_ms` mapper receives numeric driver values and yields valid `Date` instances.

#### Scenario: PRAGMA table_info on user.createdAt returns INTEGER

- **WHEN** an operator runs `PRAGMA table_info(user)` against the production D1 database
- **THEN** the row for `createdAt` SHALL report `type = 'INTEGER'`
- **AND** the same SHALL hold for `updatedAt` and `banExpires`
- **AND** `PRAGMA table_info(account)` SHALL likewise report `INTEGER` for `createdAt` and `updatedAt`

#### Scenario: Typeof stored values is integer

- **WHEN** an operator runs `SELECT typeof(createdAt), typeof(updatedAt) FROM user`
- **THEN** every row SHALL report both columns as `'integer'`
- **AND** no row SHALL report either column as `'text'` or `'real'`

#### Scenario: New user insert stores integer

- **WHEN** a new user signs in through the Google OAuth flow and better-auth inserts a `user` row
- **THEN** `SELECT typeof(createdAt) FROM user WHERE id = <new_id>` SHALL return `'integer'`
- **AND** the inserted value SHALL be a Unix epoch millisecond count within the same second that the insert occurred

### Requirement: Admin Member List Tolerates Timestamp Drift

The admin member list endpoint at `GET /api/admin/members` SHALL emit a valid ISO 8601 string or explicit `null` for each row's `createdAt` and `updatedAt`, even when the underlying driver value is a string, a float, or a value that drizzle's timestamp mapper cannot parse. The endpoint SHALL NOT propagate a `RangeError` from `Date.prototype.toISOString()` to the HTTP response under any row content.

#### Scenario: Handler returns 200 with ISO strings when driver value is float-like text

- **WHEN** any row in the `user` table holds `createdAt` stored as the TEXT literal `"1776332449872.0"` (a legacy drift value)
- **AND** an admin requests `GET /api/admin/members`
- **THEN** the endpoint SHALL respond with HTTP 200
- **AND** that row's `createdAt` in the JSON response SHALL be the ISO 8601 string equivalent of the millisecond epoch `1776332449872`

#### Scenario: Handler returns 200 with null when a timestamp is unparseable

- **WHEN** any row holds `createdAt` as `NULL`, a non-numeric string such as `"not-a-date"`, or a negative / zero epoch
- **AND** an admin requests `GET /api/admin/members`
- **THEN** the endpoint SHALL respond with HTTP 200
- **AND** that row's `createdAt` in the JSON response SHALL be `null`
- **AND** the endpoint SHALL NOT throw `RangeError: Invalid time value`

#### Scenario: Handler returns 200 with ISO strings when driver value is a valid epoch number

- **WHEN** every row holds `createdAt` and `updatedAt` as integer milliseconds consistent with drizzle's `timestamp_ms` declaration
- **AND** an admin requests `GET /api/admin/members`
- **THEN** the endpoint SHALL respond with HTTP 200 and valid ISO 8601 strings for both fields
- **AND** the response format SHALL be identical to the drift-tolerant path, so downstream consumers are unaware which path produced the value

### Requirement: Table Rebuild Migration Preserves Rows And Foreign Keys

Any migration that corrects the storage affinity of better-auth tables SHALL preserve every existing `user` and `account` row (identified by primary key `id`) and SHALL leave every existing foreign key reference valid after the rebuild.

#### Scenario: Row count preserved

- **WHEN** an operator runs the affinity-correction migration against a D1 database containing `N` rows in `user` and `M` rows in `account`
- **THEN** after the migration commits, `SELECT count(*) FROM user` SHALL return `N`
- **AND** `SELECT count(*) FROM account` SHALL return `M`

#### Scenario: Foreign keys pass integrity check

- **WHEN** the migration reaches the integrity-check step
- **THEN** `PRAGMA foreign_key_check` SHALL return zero rows
- **AND** the migration SHALL only commit after that check passes

#### Scenario: Admin id remains mapped to same email

- **WHEN** the migration completes on a D1 database where an Admin user was identified by email `admin@example.com` and id `X` before the migration
- **THEN** `SELECT email FROM user WHERE id = 'X'` SHALL return `'admin@example.com'`
- **AND** the Admin's session SHALL remain valid without requiring re-login
