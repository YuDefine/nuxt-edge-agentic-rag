## ADDED Requirements

### Requirement: Web and MCP accepted paths SHALL use Workers AI-generated answers

The system SHALL execute accepted-path answer generation through Workers AI for both the Web chat entrypoint and the MCP ask entrypoint. The shared answering core SHALL keep the existing retrieval, citation, and governance flow, but SHALL stop treating fallback answer generation as the production answer path for accepted requests.

#### Scenario: Web accepted path returns a Workers AI answer

- **WHEN** a Web chat request reaches an accepted path
- **THEN** the system SHALL generate the final answer through Workers AI
- **AND** the response SHALL preserve the citation payload required by the existing Web chat contract

#### Scenario: MCP accepted path returns a Workers AI answer

- **WHEN** an MCP ask request reaches an accepted path
- **THEN** the system SHALL generate the final answer through Workers AI
- **AND** the response SHALL preserve the citation payload required by the existing MCP contract

### Requirement: Judge paths SHALL use Workers AI without splitting Web and MCP governance

The system SHALL execute judge-path decisions through Workers AI while preserving one shared governance core for Web and MCP. The system SHALL keep the existing decision-path semantics (`judge_pass`, `judge_pass_refuse`, `self_correction_retry`, `self_correction_refuse`) instead of introducing channel-specific judgment rules.

#### Scenario: Judge pass path uses Workers AI judgment

- **WHEN** retrieval falls below the direct-answer threshold but remains eligible for judge evaluation
- **THEN** the system SHALL obtain the judgment through Workers AI
- **AND** the system SHALL continue to use the existing shared decision-path semantics

##### Example: judge-pass baseline path

- **GIVEN** a fixed sample query that does not meet the direct-answer threshold and does meet the judge-entry threshold
- **WHEN** the sample is executed through the shared answering core
- **THEN** the recorded decision path is `judge_pass`
- **AND** the final answer is generated by Workers AI

### Requirement: Accepted-path verification SHALL be reproducible and evidence-backed

The system SHALL provide a reproducible accepted-path verification flow for both Web and MCP. The verification flow SHALL include fixed samples, rerunnable smoke steps, and evidence that can be matched to real Workers AI or AI Gateway activity for the same run.

#### Scenario: Fixed sample run verifies Web and MCP accepted paths

- **WHEN** the project runs the approved fixed sample set
- **THEN** the run SHALL include at least one accepted-path verification for Web and one for MCP
- **AND** the run SHALL retain evidence that can be matched to the corresponding Workers AI activity

#### Scenario: Accepted-path verification covers direct and judge-assisted answers

- **WHEN** the fixed sample set is prepared for release or review
- **THEN** it SHALL cover at least one `direct_answer` path and one `judge_pass` path
- **AND** each covered path SHALL be rerunnable without changing the capability contract

### Requirement: Cost and latency claims SHALL distinguish measured baselines from scenario estimates

The system SHALL derive its published cost and latency baseline from a limited set of real Workers AI sample runs. Any extrapolated numbers based on assumptions or mock data SHALL be labeled as scenario estimates rather than reported as measured production totals.

#### Scenario: Baseline and estimate are labeled separately

- **WHEN** a report, poster, or proposal presents answering cost or latency numbers
- **THEN** the measured fixed-sample baseline SHALL be identified separately from scenario-based extrapolations
- **AND** the extrapolated values SHALL not be described as full production measurements
