# 專題口試答辯準備文件

> **專題標題**：基於邊緣原生架構之混合式檢索增強生成系統設計與實作：以中小企業知識庫為例
>
> **使用說明**：本文件為口試準備用，包含可能被挑戰的問題與防禦策略。建議熟悉所有論點，並準備資料佐證。
>
> **答辯口徑切分原則**：
>
> - **已實作主軸**：`current-version-only`、`citationId` 回放、三級權限治理、遮罩後查詢日誌、signed upload 與文件版本生命週期。
> - **下一階段擴充**：`pdf` / `docx` / `xlsx` / `pptx` 的 canonical snapshot ingestion、legacy Office conversion、音檔 / 影片 transcript pipeline、條件式 cloud fallback。
> - **避免說法**：不要把「架構保留能力」說成「目前已完成自動路由」，也不要把估算數字（例如 70/30、80%、<50ms）當成這一版已驗證的 production 統計。

## 核心論述框架

```
本專題答辯時建議抓三個差異化主軸：

┌─────────────────────────────────────────────────────────────┐
│  1️⃣ Edge-native managed RAG shell                           │
│     • Cloudflare Workers + D1 + R2 + KV + AI Search          │
│     • 重點是低維運、清楚治理邊界、可部署可驗證                    │
│     • 不把所有 AI 路由都說成這一版已完整落地                     │
├─────────────────────────────────────────────────────────────┤
│  2️⃣ 可驗證的 current-version-only + citation replay         │
│     • 只用 active / indexed / current 證據回答                 │
│     • `citationId` 可回放，不依賴供應商 chunk id               │
│     • 評審最容易感受到的是「可追溯」與「不亂答」                  │
├─────────────────────────────────────────────────────────────┤
│  3️⃣ 可擴充的 canonical snapshot ingestion                    │
│     • 現階段核心驗收仍以 md/txt/預轉 Markdown 為主              │
│     • 下一階段優先擴到 pdf/docx/xlsx/pptx                     │
│     • legacy Office 與音影音分別走 conversion / transcript     │
└─────────────────────────────────────────────────────────────┘
```

## 一、架構選型挑戰

### 1.1 為什麼選 Serverless Edge？

**挑戰問題**：

> 「為什麼不用傳統 VPS 或 Container（Docker/K8s）部署？Serverless Edge 架構的優勢在哪？」

**防禦論述**：

| 面向          | 傳統 VPS / Container            | Serverless Edge (Cloudflare)           |
| ------------- | ------------------------------- | -------------------------------------- |
| **維運負擔**  | 需管理 OS、安全更新、憑證、監控 | 低維運，平台處理多數基礎維護           |
| **成本模型**  | 固定月費（閒置也需付費）        | 按量計費，空閒成本較低                 |
| **擴展**      | 需手動配置 Auto Scaling         | 自動擴展，無需管理                     |
| **延遲**      | 固定機房位置，跨區用戶延遲較高  | 全球節點多，就近回應，體感延遲通常較低 |
| **冷啟動**    | Container 冷啟動通常較明顯      | Workers 冷啟動通常較短                 |
| **可用性**    | 單點故障風險，需自建 HA         | 平台提供高可用基礎設施                 |
| **DDoS 防護** | 需額外購買或設定                | 內建於平台                             |

**關鍵論點**：

> 「對於中小企業而言，Serverless 架構的最大價值是『顯著降低對專職 DevOps 的依賴』。傳統架構需要更多伺服器管理、安全更新與監控設定；Serverless 把較多基礎維護交給平台處理，讓有限的人力更集中在業務邏輯開發。」

**答辯可用的架構直覺示意**（不要說成專題實測）：

```
台灣用戶查詢延遲比較（假設傳統伺服器在美西）：
┌─────────────────┬────────────────┬───────────────┐
│ 架構             │ 網路延遲        │ 總回應時間     │
├─────────────────┼────────────────┼───────────────┤
│ 傳統 VPS（美西） │ 跨區延遲較高    │ 體感通常較慢   │
│ Edge（在地節點） │ 網路距離較短    │ 體感通常較快   │
└─────────────────┴────────────────┴───────────────┘
→ 這張表只是說明 edge 可能帶來的延遲優勢，不是本專題正式 benchmark 結果
```

**適用場景說明**：

> 「Edge-Native 並非萬能。若系統需要大量 CPU 運算（如影片轉檔）、長時間執行任務（>30秒）、或複雜資料庫交易，傳統架構更適合。本專題的 RAG 查詢屬於『短時、高頻、低運算』特性，正好符合 Edge 的設計理念。」

**追問預備**：

- Q: 「CPU 時間限制 10-50ms 夠用嗎？」
- A: 「RAG 查詢的 CPU 密集運算主要在模型與檢索服務側，應用層本身主要負責驗證、組裝與回應流程控制。答辯時不建議背固定毫秒數，而是強調這類短時請求通常更符合 Edge 的設計特性。」

### 1.2 為什麼主張 Hybrid Managed RAG，而不是每次都直接打純雲端 LLM？

**挑戰問題**：

> 「為什麼不直接全部用 GPT 5.4 或 Gemini 3.1 Pro / Gemini 3 Flash？搞這麼複雜的 Hybrid 架構有什麼意義？」

**防禦論述**：

| 面向         | 純雲端 LLM（每次都打外部模型） | Hybrid Managed RAG（本專題主張）                                |
| ------------ | ------------------------------ | --------------------------------------------------------------- |
| **主幹流程** | 每次查詢都直接依賴外部模型     | 先走受治理的檢索、驗證、拒答與引用流程                          |
| **成本彈性** | 每次都付費，成本完全綁供應商   | 可依需求決定是否開啟 cloud fallback                             |
| **隱私治理** | 所有問題都送至第三方           | 可按資料等級與環境策略決定是否外送                              |
| **答辯重點** | 容易被帶成模型比拼             | 可把主軸放在 `current-version-only`、replay、RBAC、auditability |
| **擴充性**   | 供應商變動時調整成本高         | 可保留 cloud fallback、conversion、transcript 等後續插槽        |

**答辯重點**：

> 不需要背固定價格或固定分流比例。這一版真正要守住的說法是：系統先建立可治理、可拒答、可引用回放的 RAG 主幹，再視需求決定是否開啟 cloud fallback，而不是把「70/30 自動切換」當成已驗證成果。

**關鍵論點**：

> 「Hybrid Managed RAG 的核心價值在於『先治理、後模型』。也就是先決定哪些證據可用、何時該拒答、引用如何回放，再去決定是否需要外部模型協助。」

**追問預備**：

- Q: 「邊緣 / 雲端模型品質怎麼比？」
- A: 「答辯時不建議把焦點放成單一模型 PK。這一版真正完成的是治理主幹：`current-version-only`、引用回放、拒答與權限邊界。未來若接入實模型，邊緣模型與 cloud fallback 只是接在這條主幹後面的執行層，不改變核心契約。」

## 二、技術選型挑戰

### 2.1 為什麼選 Cloudflare 而非 AWS/GCP/Azure？

**挑戰問題**：

> 「大企業都用 AWS，為什麼選 Cloudflare 這種相對小眾的平台？」

**防禦論述**：

> 這一段是選型理由，不是本專題自己逐項跑完的雲平台 benchmark。

| 面向           | AWS Lambda@Edge        | Cloudflare Workers |
| -------------- | ---------------------- | ------------------ |
| **冷啟動**     | 通常較明顯             | 通常較短           |
| **免費額度**   | 入門門檻較高           | 有免費額度可做原型 |
| **向量資料庫** | 需另購 (Pinecone等)    | Vectorize 內建     |
| **AI 推論**    | 需 SageMaker           | Workers AI 內建    |
| **部署複雜度** | 高（IAM、VPC、Lambda） | 低（NuxtHub 一鍵） |
| **開發體驗**   | 較差（CloudWatch）     | 較佳（本地模擬）   |
| **定價透明度** | 複雜（多項計費）       | 簡單（按請求計費） |

**關鍵論點**：

> 「對於本專題的規模與需求，Cloudflare 提供了更簡單的開發體驗與更低的入門門檻。AWS 的優勢在於企業級功能與生態系整合，但對於 RAG 原型驗證來說是 overkill。此外，Cloudflare 的 AI 生態（Workers AI + Vectorize）專為這類場景設計。」

---

### 2.2 為什麼用 D1 而非 PostgreSQL？

**挑戰問題**：

> 「D1 是 SQLite，功能比 PostgreSQL 少很多，10GB 上限也太小了吧？」

**防禦論述**：

| 面向         | PostgreSQL (Neon/Supabase) | D1 (SQLite)            |
| ------------ | -------------------------- | ---------------------- |
| **延遲**     | 需額外網路往返             | 與邊緣執行環境整合度高 |
| **功能**     | 完整 SQL、JSON、Full-text  | 基本 SQL               |
| **擴展性**   | 垂直擴展佳                 | 水平擴展（多個 DB）    |
| **成本**     | $5-25/月起                 | 可先從較低基礎成本起步 |
| **容量**     | 通常 10GB+                 | 10GB 硬限制            |
| **適用場景** | 複雜查詢、交易處理         | 簡單 CRUD、讀多寫少    |

**10GB 限制分析**：

> 單就本專題現階段資料型態來看，D1 主要承擔結構化 metadata、`source_chunks` 與查詢日誌，尚未進入大型交易或海量分析場景；因此 10GB 對目前答辯範圍內的中小型知識庫仍可作為可接受起點。但正式可承載量仍會受到 chunk 策略、保留週期與後續功能擴充影響，答辯時不建議把單一容量試算說成實測上限。

**關鍵論點**：

> 「本專題的資料模型相對簡單（文件、區塊、使用者、日誌），不需要 PostgreSQL 的進階功能。D1 的優勢在於與 Workers 的整合度高，這對 RAG 這類短時查詢流程是有利的。若未來需求超過限制，可遷移至 Turso（LibSQL，相容 SQLite）。」

---

### 2.3 為什麼用 Better Auth 而非 Auth0/Clerk？

**挑戰問題**：

> 「Auth0 是業界標準，為什麼用這個比較新的框架？」

| 面向          | Auth0 / Clerk           | Better Auth      |
| ------------- | ----------------------- | ---------------- |
| **定價**      | $23-35/月起（MAU 計費） | 開源免費         |
| **Edge 相容** | 部分支援                | 原生支援 Workers |
| **自託管**    | 不支援                  | 完全自託管       |
| **資料控制**  | 資料在第三方            | 資料在自己的 D1  |
| **Passkey**   | 需額外付費              | 內建 Plugin      |
| **客製化**    | 有限                    | 完全可控         |

**關鍵論點**：

> 「對於專題而言，Better Auth 提供了足夠的功能且成本為零。同時，自託管模式讓我們完全掌控使用者資料，這對企業知識庫系統的隱私合規很重要。若企業需要 SOC2 認證等合規需求，報告中已說明可評估 Auth0 Enterprise。」

### 2.4 為什麼文件格式採分級支援，而不是一開始就宣稱全部都能 ingest？

**挑戰問題**：

> 「既然是企業知識庫，為什麼不直接說 `.doc`、`.docx`、`.xls`、`.xlsx`、`.ppt`、`.pptx`、音檔、影片全部都支援？」

**防禦論述**：

| 層級                             | 格式                               | 答辯口徑                                                                                 |
| -------------------------------- | ---------------------------------- | ---------------------------------------------------------------------------------------- |
| **核心驗收**                     | `.md`, `.txt`, 預先轉 Markdown     | 目前最穩定，直接對應既有 replay 與引用驗證流程                                           |
| **下一階段優先擴充**             | `.pdf`, `.docx`, `.xlsx`, `.pptx`  | 先抽成 canonical text snapshot，再進既有 `normalized_text_r2_key` / `source_chunks` 契約 |
| **後續 conversion / transcript** | `.doc`, `.xls`, `.ppt`, 音檔, 影片 | 不應硬塞進目前同步 request path，應分別走 conversion 或 transcript pipeline              |

**關鍵論點**：

> 「我們不是只看『能不能上傳』，而是看『上傳後能不能穩定引用、回放、稽核』。因此最佳實踐不是一次把所有副檔名都打開，而是先把 PDF 與 OOXML 納入 canonical snapshot ingestion，再把 legacy Office 與媒體轉錄獨立成第二階段。」

**追問預備**：

- Q: 「那 `.doc`、`.xls`、`.ppt` 呢？」
- A: 「這三種是 legacy binary Office。最佳實踐不是在目前的 Workers request path 內硬 parse，而是先做 conversion boundary，轉成現代 Office 或文字稿，再進既有引用與回放契約。」
- Q: 「音檔跟影片是不是也能做？」
- A: 「可以，但難度顯著更高。因為那已不是文件解析，而是 transcript pipeline，通常要獨立處理上傳、轉錄、校閱與時間戳對應，不適合和文件 ingestion 混成同一波。」

```

4. **供應商風險**：API 變更、限流、定價調整都會直接影響純雲端方案

## 四、RAG 技術挑戰

### 4.1 Self-Correction 真的有效嗎？

**挑戰問題**：

> 「你說 Self-Correction 可以提升精度，有什麼根據？」

**學術依據**：

- [7] Liu et al., "Can LLMs Correct Themselves? CorrectBench"
- [6] "RAG-Gym: Systematic Optimization of Language Agents for RAG"

**機制說明**：

```
傳統 RAG：
Query → Retrieve → Generate（單次，可能檢索錯誤）

Self-Correction RAG：
Query → Retrieve → 評估信心度
  ├─ 高於門檻 → 直接回答
  └─ 低於門檻 → 重寫 Query → 二次 Retrieve → 再決定是否回答
```

**實驗建議**：

- 準備 50 題測試集（含易、中、難）
- A/B 測試：有無 Self-Correction 的回答品質比較
- 觀察重點：低信心查詢是否比單輪檢索更容易找到可引用證據；不要先宣稱固定提升幅度

**誠實承認限制**：

> 「Self-Correction 並非萬能。若知識庫本身沒有相關資料，重試也無法產生正確答案。此外，重試通常會增加延遲，因此需要在品質與速度間取捨。」

---

### 4.2 向量檢索的準確度如何保證？

**挑戰問題**：

> 「向量相似度高就代表答案正確嗎？」

**誠實回答**：

> 「向量相似度衡量的是『語義相近』而非『答案正確』。高相似度的文件可能語義相關但不是正確答案。」

**緩解措施**：

1. **多文件交叉驗證**：檢索 Top-5 而非 Top-1
2. **信心度閾值**：低於閾值時明確告知使用者
3. **引用來源**：回答附帶來源連結
4. **拒答機制**：知識庫外的問題明確拒答

## 五、安全性挑戰

### 5.1 Passkey 的普及度問題

**挑戰問題**：

> 「Passkey 還很新，很多使用者不會用怎麼辦？」

**現況資料**（答辯前再核對官方支援矩陣）：

- 主流瀏覽器皆已支援（Chrome, Safari, Firefox, Edge）
- iOS 16+ / Android 9+ 原生支援
- Windows Hello 已普及於企業筆電
- Apple/Google 帳號已可同步 Passkey

**降級策略**：

```
登入優先順序：
1. Passkey（最佳體驗）→ 指紋/臉部辨識
2. Google OAuth（熟悉的一鍵登入）
3. [未來可加] Email Magic Link（仍為無密碼）
```

**關鍵論點**：

> 「無密碼不等於只有 Passkey。Google OAuth 對大多數使用者來說已足夠熟悉，Passkey 則提供更好的安全性與操作體驗。兩者並存，可兼顧採用門檻與安全性。」

---

### 5.2 資料送入 LLM 的隱私問題

**挑戰問題**：

> 「ERP 資料可能包含敏感資訊，送給 OpenAI 安全嗎？」

**設計上的隱私保護**：

1. **治理優先**：可依資料等級與風險決定哪些查詢留在自控回答層、哪些才允許外送
2. **資料脫敏**：上傳文件時可預處理，移除個資
3. **選擇性切換**：可設定某些分類永不使用雲端 LLM
4. **供應商治理**：外部雲端模型僅限核可名單，且需於答辯前再次確認官方資料政策

**敏感度分級設計**：

```
┌────────────────┬─────────────────┐
│ 敏感度          │ LLM 使用策略     │
├────────────────┼─────────────────┤

## 八、開發過程與工具選型

### 8.1 為什麼採用 Spec-Driven Development？

**挑戰問題**：

> 「為什麼不直接用 AI 寫程式就好？搞什麼規格驅動？」

**防禦論述**：

**傳統 AI 輔助開發的問題**：

```
「Vibe Coding」模式：
模糊 Prompt → AI 猜測意圖 → 產生可能錯誤的程式碼 → 人工修正 → 反覆來回
```

**Spec-Driven Development 解決方案**：

```
spectra（基於 OpenSpec）流程：
Discuss（討論）→ Propose（提案 / 規格 + 任務）→ Apply（實作）→ Archive（歸檔）
       ↓                    ↓                           ↓              ↓
  對齊需求與設計    產出 proposal/spec/design/tasks   AI 依規格實作    封存完成變更
```

**關鍵論點**：

> 「Spec-Driven Development 的核心是『規格即真理』。AI 不再猜測我們要什麼，而是依據明確的規格文件生成程式碼。這大幅減少了『AI 產生的程式碼需要人工修正』的時間。」

**OpenSpec 的核心理念**：

> 「AI coding assistants are powerful but unpredictable when requirements live only in chat history.」spectra 以視覺化方式將此理念落地，把每一次變更組織為包含 proposal、spec、design 與 tasks 的獨立資料夾。

### 8.3 開發效率的保守說法

**挑戰問題**：

> 「說節省 50-60% 時間，有什麼根據？」

**建議回答**：

> 這一題不建議把答辯重心放在精確工時百分比，因為本專題沒有把工時研究做成正式實驗。比較安全的說法是：工具鏈確實減少了查文件、試錯與來回修正，但這裡屬於開發經驗觀察，不是論文式量化結論。

**可以強調的三點**：

- spectra 讓需求、設計與任務先對齊，減少 AI 憑感覺猜需求。
- MCP 讓 AI 取得較新的框架知識，減少過時語法造成的反覆修正。
- 規格、實作、驗證串在同一條流程上，降低上下文切換成本。

### 8.5 MCP 解決的核心問題

**挑戰問題**：

> 「MCP 到底解決什麼問題？沒有 MCP 不能開發嗎？」

**AI 訓練資料過時的問題**：

```
傳統 AI 輔助的痛點：

User: 「幫我寫一個 Nuxt 的 useFetch」

AI (訓練資料是 Nuxt 2/3):
「export default {
  async asyncData({ $axios }) { ... }  ← 這是 Nuxt 2 語法！
}」

User: 「這不對，Nuxt 4 不是這樣寫」

AI: 「抱歉，讓我修正...」（還是給錯的）

→ 來回 3-5 次才能得到正確答案
```

**MCP 的解決方案**：

```
有 MCP 的情況：

User: 「幫我寫一個 Nuxt 的 useFetch」

Claude Code: [透過 MCP 查詢 Nuxt 4 文件]

AI:
「const { data, status } = await useFetch('/api/data')」
   ↑ 這是正確的 Nuxt 4 語法！

→ 一次就對
```

**較穩妥的說法**：

> MCP 真正解決的是「AI 用過時知識回答新框架問題」這件事。答辯時可以說它明顯減少了語法試錯與反覆修正，但不要把它講成做過嚴格統計後的錯誤率研究。

| 11   | Admin 進入「訪客權限 Dial」設定頁，將 dial 切為 `browse_only`，以 Guest 重登                                                       | Guest 進入 `/chat` 看到「此環境目前僅開放瀏覽，不可問答」提示；POST `/api/chat` 被 server 拒絕回 403 | A08（含 B16 訪客 dial）            |

### 備援情境

若示範過程任一步驟失敗，備援展示：

1. **Rollback 展示**：引用報告附錄 C（部署與災難復原）或 `docs/deployment/ROLLBACK.md`，示範 `wrangler rollback` 單步驟還原 Workers bundle
2. **Restricted 隔離**：切換至無 `knowledge.restricted.read` scope 之 MCP token，呼叫 `askKnowledge` 查詢 restricted-only 內容，驗證 `refused = true` 且不洩漏存在性（TC-17）
3. **高風險輸入治理**：於 Chat 貼上模擬 API token 字串，驗證 `messages.content_redacted` 只存遮罩版本、`query_logs.status = 'blocked'`（TC-15、A11）

### 示範後清理

結束示範後，於 Staging 環境：

1. 撤銷示範中建立之 Member（降為 Guest 或 `/api/admin/members/[userId]/role` demote）
2. 重設訪客權限 Dial 為 `same_as_member`
3. 保留 `query_logs` 與 `citation_records` 180 天作為稽核證據（符合 §2.4.1.4 保留期限）
4. 依需要手動 archive 示範用文件
